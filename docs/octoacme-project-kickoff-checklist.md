# OctoAcme Project Kickoff Checklist

## Purpose
Use this checklist during project initiation to ensure all roles are assigned, aligned, and ready to execute. This ensures clear accountability and smooth collaboration from day one.

---

## Pre-Kickoff (1 week before)

### Stakeholder Alignment
- [ ] Identify and confirm Stakeholder/Sponsor for the project
- [ ] Confirm Stakeholder availability for kickoff meeting and ongoing syncs
- [ ] Share Project One-pager with Stakeholder for review and sign-off
- [ ] Clarify decision authority and approval process with Stakeholder

### Team Formation
- [ ] Confirm Project Manager assigned and available
- [ ] Confirm Product Manager assigned and available
- [ ] Confirm Technical Lead assigned and available
- [ ] Confirm QA/Testing Lead assigned and available
- [ ] Confirm Scrum Master assigned and available
- [ ] Identify and confirm core developer team members
- [ ] Confirm all team members' availability for sprint cadence

### Preparation
- [ ] Project One-pager completed (problem, goal, success metrics)
- [ ] Initial backlog items drafted
- [ ] High-level timeline and milestones defined
- [ ] Risk register started with known risks
- [ ] Communication plan drafted (cadence, channels, stakeholders)

---

## Kickoff Meeting (1–2 hours)

### Agenda Items

#### 1. Welcome & Business Context (15 min)
- [ ] Stakeholder/Sponsor provides business context and objectives
- [ ] Project Manager reviews Project One-pager and timeline
- [ ] Success metrics and definitions of "done" reviewed
- **Owner:** Project Manager / Stakeholder/Sponsor

#### 2. Role Clarifications & Responsibilities (15 min)
- [ ] Review OctoAcme roles and personas (use `octoacme-roles-and-personas.md`)
- [ ] Confirm each role assignment and introduce team members
- [ ] Review Roles Interaction Matrix (use `octoacme-roles-interaction-matrix.md`)
- [ ] Clarify decision authority for key decisions
- **Owner:** Project Manager

#### 3. Technical Vision & Approach (15 min)
- [ ] Technical Lead presents high-level architecture and approach
- [ ] Technical Lead highlights technical risks and dependencies
- [ ] Discuss testability, quality gates, and deployment approach
- [ ] QA/Testing Lead confirms test strategy alignment
- **Owner:** Technical Lead / QA Lead

#### 4. Delivery Timeline & Plan (15 min)
- [ ] Project Manager reviews milestones and sprint schedule
- [ ] Discuss dependencies and integration points
- [ ] Review release timeline and go-live readiness criteria
- [ ] Confirm team capacity and resource availability
- **Owner:** Project Manager / Scrum Master

#### 5. Communication & Ceremonies (15 min)
- [ ] Scrum Master reviews sprint cadence and ceremony schedule
- [ ] Review status reporting format and frequency
- [ ] Confirm stakeholder update schedule and format
- [ ] Clarify escalation paths (use `octoacme-risks-and-communication.md`)
- **Owner:** Scrum Master / Project Manager

#### 6. Q&A & Next Steps (10 min)
- [ ] Open floor for questions
- [ ] Confirm sprint 0 activities and first sprint start date
- [ ] Assign action items and owners
- **Owner:** Project Manager / Scrum Master

---

## Post-Kickoff (within 1 week)

### Documentation & Setup
- [ ] Project repository / board created and shared with team
- [ ] All team members added to project repository with appropriate access
- [ ] Communication channels (Slack, email, meetings) set up and confirmed
- [ ] Initial backlog items created with acceptance criteria
- [ ] Definition of Done (DoD) finalized and shared
- [ ] Risk register created and shared
- [ ] Project charter / One-pager added to repo

### Schedule Confirmation
- [ ] Daily standup time scheduled and calendar invites sent
- [ ] Weekly execution sync scheduled (team + PM)
- [ ] Weekly PM/PdM sync scheduled
- [ ] Sprint planning, review, and retrospective times confirmed
- [ ] Stakeholder update meetings scheduled (weekly, bi-weekly, or monthly as agreed)
- [ ] Milestone review dates scheduled

### Role-Specific Onboarding
- [ ] **Developers**: Repository access, branching strategy, PR process, local setup instructions
- [ ] **QA/Testing Lead**: Test environment access, test data setup, testing tools, acceptance criteria template
- [ ] **Technical Lead**: Architecture docs, design review process, code review guidelines, technical risk tracking
- [ ] **Product Manager**: Backlog management, analytics/measurement tools, customer/user research access
- [ ] **Project Manager**: Project board setup, risk register ownership, stakeholder list, communication templates
- [ ] **Scrum Master**: Sprint board configuration, ceremony facilitation approach, impediment tracking

### First Sprint Activities (Sprint 0 or Sprint 1)
- [ ] Confirm sprint goal and backlog items for first sprint
- [ ] Conduct estimation session or backlog refinement
- [ ] All team members complete environment setup and can execute a test build
- [ ] QA/Testing Lead confirms test environment and can execute smoke tests
- [ ] Project Manager confirms all team members can access project board and documentation
- [ ] Scrum Master runs first standup to confirm ceremony flow
- [ ] Technical Lead conducts first design review or architecture discussion

## Roles & Responsibilities Check-In

Use this section to confirm each role is ready to execute:

### ✅ Stakeholder/Sponsor Ready When:
- [ ] Business objectives and success metrics clearly defined
- [ ] Authority and approval process clarified
- [ ] Availability confirmed for monthly status updates and milestone reviews
- [ ] Escalation path understood

### ✅ Project Manager Ready When:
- [ ] Project plan created with timeline and milestones
- [ ] Risk register initiated and shared
- [ ] Communication plan defined
- [ ] Team roles and responsibilities confirmed
- [ ] Status reporting format agreed with stakeholders
- [ ] All team members onboarded

### ✅ Product Manager Ready When:
- [ ] Backlog prioritized with acceptance criteria
- [ ] Success metrics and measurement approach defined
- [ ] User research and market context shared with team
- [ ] Feature roadmap communicated to team
- [ ] Weekly PM/PdM sync scheduled

### ✅ Technical Lead Ready When:
- [ ] High-level architecture and technical approach defined
- [ ] Technical risks identified and mitigation plans started
- [ ] Code review process and standards documented
- [ ] Design review cadence confirmed
- [ ] Development environment and tools set up

### ✅ QA/Testing Lead Ready When:
- [ ] Test strategy and test plan drafted
- [ ] Acceptance criteria template created
- [ ] Test environment set up and confirmed
- [ ] Definition of Done quality criteria defined
- [ ] Testing tools and automation approach documented

### ✅ Developers Ready When:
- [ ] Repository access confirmed
- [ ] Development environment set up and test build successful
- [ ] Branching strategy and PR process understood
- [ ] First sprint backlog items understood with clear acceptance criteria
- [ ] Daily standup time confirmed

### ✅ Scrum Master Ready When:
- [ ] Sprint board configured and all team members have access
- [ ] Sprint length, ceremony times, and locations confirmed
- [ ] Retrospective format and action item tracking process defined
- [ ] Impediment escalation process understood
- [ ] First sprint ceremonies scheduled

## How to Use This Checklist

- **Print or copy** this checklist for each new project
- **Assign ownership**: Project Manager typically owns the overall checklist, but may delegate items to role leads
- **Complete pre-kickoff items** 1 week before the kickoff meeting
- **Run through the kickoff agenda** in the order listed, but adjust based on team size and project complexity
- **Complete post-kickoff items** within 1 week of the kickoff meeting
- **Keep this checklist in your project repository** for future reference and continuous improvement
- **Revisit after first sprint**: Check if any items need adjustment for future projects
